# 全栈 CS 名词学习路线与写作计划

这份文件是项目的主施工图。

---

## 一句话定位

这是一套面向 AI 协作开发时代的“软件开发概念地图”：帮助读者看懂软件从想法到交付的全过程，并能把模糊想法写成人能评审、AI 能执行、结果能验收的软件规格。

---

## 核心判断

### 不是传统全栈教程

这里的读者可能已经能用 AI 生成代码，但仍然会被名词卡住：不知道 REST 为什么出现，不知道 JWT 和 OAuth 的边界，不知道 CI/CD 和部署有什么区别，不知道 spec 里什么时候应该写数据库、缓存、权限、错误处理。

所以本项目优先解决“认知断点”，而不是训练手写代码熟练度。

### 不是百科词典

百科词典按字母或领域罗列名词，但读者很难知道先读什么、读完能干什么。

本项目按“软件从想法到交付”的过程组织。每个名词都要被放回一个真实环节里：它在什么时候出现，解决什么问题，和前后环节有什么关系。

---

## 目标读者

### 第一类：AI 编程新手

他们会和 ChatGPT、Claude、Cursor、Codex、Copilot 等工具对话，但经常不知道模型生成的技术词是什么意思。这个手册帮助他们把“会问 AI”升级成“会给 AI 写清楚任务”。

### 第二类：非传统工程背景的人

产品、设计、运营、独立开发者、创业者，可能不想成为纯程序员，但需要和工程系统打交道。这个手册帮助他们理解软件开发语言。

### 第三类：想补齐全栈认知的开发者

他们可能熟悉某个局部，比如前端或 Python，但对部署、安全、系统设计、AI 应用工程缺少整体地图。这个手册帮助他们补上下游视角。

---

## 最终读完应该获得什么

读者读完后，不一定能独立手写一个复杂系统，但应该能做到：

1. 听懂常见软件开发名词，不被术语打断思考
2. 知道一个软件从想法、规格、开发、测试、部署到运营的大致流程
3. 能把模糊需求拆成页面、接口、数据、权限、错误处理、验收标准
4. 能读懂 AI 生成代码时提到的工具、框架、协议和工程概念
5. 能写出人能评审、AI 能执行、结果能验收的软件规格
6. 能判断某个概念是“当前必须处理”，还是“未来规模变大后再处理”

---

## 写作总原则

### 1. 先场景，后定义

不要一上来背定义。先告诉读者为什么会遇到这个词。

例如 REST 不从“Representational State Transfer”开始，而从“前端、App、小程序都需要用稳定方式调用后端能力”开始。

### 2. 先位置，后细节

每个名词都要回答：它位于软件流程的哪个位置？属于需求、前端、后端、数据、部署、安全、AI 编程，还是系统规模化问题？

### 3. 先横向对比，再纵向深入

讲一个词之前，先给出相邻概念：

- REST vs GraphQL vs gRPC
- Session vs Token vs JWT vs OAuth
- SQL vs NoSQL vs 对象存储
- Docker vs 虚拟机 vs Kubernetes
- Prompt vs Spec vs Workflow vs Agent

读者知道边界后，再讲当前主角。

### 4. 递归解释，但控制深度

解释 A 时遇到 B，要解释 B；但不要无限展开。当前笔记只补足理解 A 所需的最低概念，深度内容链接到专门笔记。

### 5. 服务可交付规格，而不是服务炫技

每篇笔记最后都要尽量回答：如果我要写一份可交付的软件规格，这个概念应该如何出现？

有些概念需要写进 spec，例如 API 路径、数据字段、权限、验收标准。
有些概念通常不写进早期 spec，例如具体缓存算法、Kubernetes 资源细节，除非项目明确需要。

### 6. 代码只展示，不运行

本项目不是代码实验课。代码块用于帮助理解语法、配置或结构，不要求执行。

### 7. 面试相关内容可以适度扩展

如果某个概念恰好也是面试高频点，可以在笔记中增加“面试常见问法”或“容易追问的点”。但扩展的目的仍然是帮助理解工程概念，而不是把手册改成刷题题库。

推荐写法：

~~~text
## 面试常见问法

- 这个概念通常会怎么问？
- 面试官真正想确认什么？
- 回答到什么深度就够了？
- 哪些说法容易显得只背过八股？
~~~

## 单篇笔记推荐模板

每篇 .ipynb 推荐使用以下 markdown cell 结构。

~~~text
# 名词标题

## 为什么需要它
从历史动机或真实场景讲起。

## 它是什么
给出清晰定义，避免堆术语。

## 它不是什么
排除常见误解。

## 和相邻概念的区别
用表格或短列表横向对比。

## 怎么使用
给出最小可理解示例。代码只展示，不运行。

## 在规格里怎么写
说明它是否应该出现在软件规格中，以及写到什么粒度。

## 常见坑
列出初学者容易误解的点。

## 延伸阅读方向
指向后续相关笔记。
~~~

---

## 文件格式规则

- 全部使用 .ipynb
- 默认只使用 markdown cell
- 所有代码都写在 markdown fenced code block 中
- 不使用可执行 code cell，除非未来某篇明确需要可运行实验
- 每篇笔记只讲一个核心名词或概念
- 文件名用中文清楚表达主题，必要时保留英文缩写

示例：

~~~text
03-网络与API契约/
├── 01-HTTP概述.ipynb
├── 02-REST：理解常见API设计风格.ipynb
└── 03-OpenAPI与Swagger：把接口写成协作文档.ipynb
~~~

## REST 的边界说明

REST 要写，因为它是理解 API 的核心名词；但不要写成“本项目的 REST API 设计”。本项目是学习手册，不是服务端应用。

REST 笔记应该重点回答：

- 为什么需要一种稳定的 API 风格

- 状态码如何表达结果
- REST 和 GraphQL、gRPC 的边界

## 质量标准

一篇笔记写完后，用下面的问题检查：

1. 读者能不能说出这个名词为什么出现？
2. 读者能不能把它放回软件流程中的正确位置？
3. 有没有解释相邻概念的区别？
4. 有没有说明它在软件规格中是否需要出现？
5. 有没有避免把展示代码写成必须运行的实验？
6. 有没有列出初学者最容易误解的点？
7. 有没有给后续笔记留下自然链接？
