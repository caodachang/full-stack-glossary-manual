# 写作规范

> 你是全栈名词手册项目的写作执行者。按本规范写每一篇笔记。

---

## 核心原则

AI 时代人不需要看懂代码，但需要看懂**全栈开发中的所有名词**——才能真正弄清流程，目的是能写出 SDD coding 的 spec。

**铁律：解释过程中引出的每一个新名词，都必须当场解释。** 读任何一篇笔记的人，不应该因为中间冒出一个没听过的词而卡住。解释 A 时说到了 B，立刻简短解释 B；解释 B 时说到了 C，继续解释 C。如果 B 内容太多，另开一篇笔记，并在原文给出交叉引用。

## 每篇笔记的结构

每个名词一篇笔记，按三步组织：

> **为什么需要它 → 是什么 → 怎么样**

| 步骤 | 回答的问题 |
|------|-----------|
| 为什么需要它 | 从历史角度讲：这个技术出现前开发者遇到什么困境，当时是怎样做的，它出现后改变了什么 |
| 是什么 | 说清这个名词 |
| 怎么样 | 它怎么工作的、跟谁配合、在整个流程的哪一环 |

## 笔记格式

全部 `.ipynb`。代码示例放在 code cell 中，通过 cell 级 metadata 指定语言，实现精确语法高亮：

```json
"metadata": {
  "vscode": {
    "languageId": "shellscript"
  }
}
```

| 代码类型 | languageId |
|---------|-----------|
| shell 命令（git/docker/uv/pip 等） | `shellscript` |
| Dockerfile | `dockerfile` |
| Docker Compose YAML | `yaml` |
| pyproject.toml | `toml` |
| .gitignore | `ignore` |
| Bash 脚本 | `shellscript` |
| 纯文本流程/示意图 | `plaintext` |

不要求可运行，只要看起来清晰易读。

## 命名

- 文件名按 [计划.md](计划.md) 中的清单，已有 `NN-` 序号前缀，直接按清单创建即可
- 子文件夹同理，按清单中的序号命名
- 例：
  ```
  01-管理工具/
  ├── 01-Git概述与基础操作.ipynb
  ├── 02-Git分支与协作.ipynb
  └── 03-GitHub协作流.ipynb
  ```

## 不要做的事

- 不要写长代码示例（AI 会写，人只需要看懂概念）
- 不要动 `.venv/`
