# 全栈名词学习笔记

一本面向 AI 协作开发时代的软件开发通识手册，目标是把模糊想法变成人能评审、AI 能执行、结果能验收的软件规格。

它不把目标定成“从零手写一个完整项目”，而是帮助读者先看懂软件开发世界里的关键名词、流程和边界：需求怎么变成规格，规格怎么拆成界面、接口、数据、部署、测试、安全和运维，AI 工具又应该在其中扮演什么角色。

> 核心目标：帮助读者从“听到开发术语会卡住”，走到“能把模糊想法写成人能评审、AI 能执行、结果能验收的软件规格”。

---

## 这个项目是什么

这是一本“从名词到可交付规格”的学习手册。

但手册会解释 REST、数据库、部署、测试、安全、RAG、Agent 等概念，因为这些词会不断出现在真实软件开发和 AI 协作开发里。读者不一定要亲手实现所有东西，但需要知道它们解决什么问题、什么时候出现、在 spec 里怎么表达。

完整路线见 [计划.md](计划.md)。

## 适合谁读

- 会用 AI 写一点代码，但经常被 Git、API、部署、数据库、认证、测试等词卡住的人
- 想从“能描述需求”进阶到“能写清楚技术规格”的产品、运营、设计、独立开发者
- 想建立全栈开发认知地图，但不想一上来就陷入框架细节的人
- 想理解 AI 编程、Agent、RAG、LLM API、SDD 这些新工作方式的人

## 阅读主线

这本手册按“软件从想法到交付”的路径组织，而不是按传统学科目录堆概念。

~~~text
电脑、文件与项目工作区
  -> 从想法到 SDD/spec
  -> Git、终端与协作工具
  -> 网络、HTTP 与 API 契约
  -> 前端界面与用户交互
  -> 数据、状态与存储
  -> 部署、运行与持续交付
  -> 测试、安全与可维护性
  -> 系统设计与规模化
  -> AI 编程、Agent 与 LLM 应用
  -> 模型基础、AI 应用生态与平台发布流程
~~~

这条路线的重点不是让读者记住所有名词，而是让读者逐渐获得一种判断力：听到一个词时，能知道它属于哪个环节、为什么存在、和其他词是什么关系、写规格时要不要提到它。

## 每篇笔记怎么写

每篇笔记只围绕一个核心名词或概念展开，尽量回答这些问题：

1. 它为什么会出现？
2. 它解决了什么问题？
3. 它到底是什么？
4. 它和相邻概念有什么区别？
5. 它在真实项目里怎么用？
6. 写 SDD/spec 时要不要描述它？怎么描述？
7. 初学者最容易误解什么？

写作时优先使用真实场景和横向对比。例如 REST 让读者理解最常见的 API 风格，并能在 spec 里写清接口约定。

## 文件格式

- 全部使用 .ipynb
- 默认只使用 markdown cell 组织正文和代码示例
- 所有代码都只作为展示示例，不要求在 notebook 中运行
- 代码示例统一写在 markdown cell 的 fenced code block 中，例如 python、bash、json、yaml
- 默认不使用可执行 code cell，避免运行报错、kernel 不匹配或语言警告

---

## 内容边界

本项目会解释软件开发中的关键技术名词，但不会把每个方向都写成完整工程教材。

- REST、GraphQL、gRPC 是 API 设计知识点，不是本项目自身要开发的接口
- Docker、Kubernetes、CI/CD 是交付和运维知识点，不是要求本项目必须容器化部署
- React、数据库、后端框架是全栈认知的一部分，不是要求本仓库变成 Web 应用
- ML/DL、RAG、Agent 是 AI 时代必须理解的概念，但笔记重点是概念地图和工程位置，不是论文推导
- 如果某个概念恰好也是面试高频点，可以适度扩展常见问法和答题边界，但不把手册写成题库
