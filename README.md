## 笔记约定

一篇笔记讲透一个 名词/概念，要讲的尽可能详细，如果某个 名词/概念 下有内容太多可以当作文件夹再往下写。每个名词按三步组织：

> **从历史的角度看为什么需要它 → 它是什么 → 它怎么使用**

### 递归解释

解释名词 A 的过程中，如果出现了名词 B，当场解释 B 是什么。解释 B 时如果又出现了 C，继续解释 C。确保读任何一个名词的笔记时，不会因为中间出现生词而卡住。

### 跨域重复处理

同一名词可能在多个领域出现（如 CORS 出现在网络协议、前端、安全三个章节）。处理规则：

- 在**核心领域**按三步（历史动机 → 是什么 → 怎么用）完整讲透
- 其他领域只讲"本领域视角下的应用"

### 先横向对比再纵向挖深

讲一个名词时，在"是什么"阶段先做横向定位——列出几个同类主流工具/概念并做简短对比，让读者知道整个品类里还有哪些选项、各自定位差异，然后再纵向深挖当前主角。例如讲 uv 先对比 pip 和 conda，再深入 uv 本身；讲 REST 先对比 GraphQL 和 gRPC，再深入 REST。

### 笔记格式

全部使用 `.ipynb`。代码示例放在 code cell 中，通过 cell 级 metadata 指定语言，实现精确语法高亮。
